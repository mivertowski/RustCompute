# RingKernel v1.1 — Multi-GPU + VynGraph Gap Closure

**Date:** 2026-04-17
**Goal:** Close 5 specific gaps blocking VynGraph NSAI integration + finalize multi-GPU with formal verification.
**Target hardware:** 2× H100 VMs on Azure (NVLink-connected)

---

## 1. Context

VynGraph is a neuro-symbolic AI (NSAI) reasoning system that plans to use RingKernel for GPU-accelerated inference chains. Audit firms run multi-step LLM reasoning workloads that require:
- Rebalancing actors across GPUs during long chains
- Live introspection for debugging stalled actors
- Hot-reloading inference rules (OWL/SHACL) without runtime restart
- PROV-O provenance attribution for reasoning steps
- Tenant isolation between audit firms

v1.1 closes these gaps and finalizes multi-GPU. All new GPU-level features will be verified on a 2×H100 Azure deployment with formal properties proven.

## 3. Gap Specifications

### 3.1 Multi-GPU Actor Migration

**Problem:** Long NSAI inference chains may saturate one GPU while others idle. No current mechanism to migrate a live persistent actor between GPUs.

**Solution:** Implement `ActorMigrator` building on existing `CheckpointManager`:

```
Source GPU:                    Target GPU:
  1. Quiesce (H2K snapshot) →
  2. Drain in-flight messages →
  3. Capture state to mapped memory
                               ← 4. Preallocate actor slot
                               ← 5. Restore state via H2K
  6. Forward pending messages →
  7. Terminate source actor
                               ← 8. Resume execution
```

**API:**
```rust
pub struct MigrationPlan {
    pub source_gpu: u32,
    pub target_gpu: u32,
    pub actor_id: ActorId,
    pub grace_period: Duration,
}

impl MultiGpuRuntime {
    pub async fn migrate_actor(&self, plan: MigrationPlan) -> Result<MigrationReport>;
    pub async fn rebalance(&self, strategy: RebalanceStrategy) -> Result<Vec<MigrationReport>>;
}

pub enum RebalanceStrategy {
    /// Move least-loaded actors to under-utilized GPU
    LoadBalance { target_imbalance: f32 },
    /// Co-locate actors that communicate frequently
    CommunicationAware,
    /// Explicit placement via user-provided plan
    Explicit(Vec<MigrationPlan>),
}
```

**Files:**
- New: `crates/ringkernel-cuda/src/multi_gpu/migration.rs` — migration orchestrator
- New: `crates/ringkernel-cuda/src/multi_gpu/mod.rs` — multi-GPU runtime facade
- Modify: `crates/ringkernel-cuda/src/persistent.rs` — add snapshot-for-migration command
- New: `crates/ringkernel-cuda/src/cuda/migration_kernels.cu` — state serialize/restore kernels

**Workaround until landed:** VynGraph can use `drain_and_recreate()` — terminate actor on source, start fresh on target. Loses in-flight messages but unblocks integration.

### 3.2 Live Actor Introspection (Streaming)

**Problem:** Debugging multi-step LLM reasoning requires seeing live actor state, not periodic snapshots. Current `ActorIntrospection` is pull-only.

**Solution:** Add streaming metrics channel via K2H queue with dedicated response type.

**API:**
```rust
pub struct LiveMetrics {
    pub actor_id: ActorId,
    pub timestamp: HlcTimestamp,
    pub queue_depth: usize,
    pub inbound_rate: f64,      // messages/sec
    pub outbound_rate: f64,
    pub latency_p50: Duration,
    pub latency_p99: Duration,
    pub state_size_bytes: u64,
    pub gpu_utilization: f32,
}

impl IntrospectionStream {
    pub fn subscribe(&self, actor_id: ActorId, interval: Duration) -> Receiver<LiveMetrics>;
    pub fn subscribe_all(&self, interval: Duration) -> Receiver<LiveMetrics>;
    pub fn unsubscribe(&self, actor_id: ActorId);
}
```

**GPU-side implementation:**
- New H2K command: `SubscribeMetrics { actor_id, interval_us }`
- New K2H response: `LiveMetrics { ... }` emitted periodically from kernel
- Per-actor metrics buffered in control block (cache-aligned, no locks)

**Files:**
- Modify: `crates/ringkernel-core/src/introspection.rs` — add `IntrospectionStream`
- Modify: `crates/ringkernel-cuda/src/persistent.rs` — add metrics H2K/K2H pair
- New: `crates/ringkernel-cuda/src/cuda/metrics_collector.cu` — GPU metric emitter

**VynGraph workaround:** Emit metrics from RustGraph layer via `runtime.kernel_metrics()` polled every N ms. Acceptable but loses the sub-millisecond detail.

### 3.3 Hot Rule Reload (Inference Rules)

**Problem:** NSAI reasoning uses OWL/SHACL rules. Updating rules currently requires restarting the GPU runtime.

**Solution:** Extend existing `HotReloadManager` with a generic `RuleSet` trait.

**API:**
```rust
pub trait RuleSet: Send + Sync {
    type Error: std::error::Error;

    /// Parse rules from source (OWL XML, SHACL Turtle, etc.)
    fn parse(source: &[u8]) -> Result<Self, Self::Error> where Self: Sized;

    /// Validate compatibility with currently loaded rules
    fn validate_transition(&self, from: &Self) -> Result<(), Self::Error>;

    /// Serialize for GPU transfer
    fn serialize_for_gpu(&self) -> Vec<u8>;
}

impl HotReloadManager {
    pub fn register_rule_set<R: RuleSet>(&mut self, name: &str, initial: R);
    pub async fn reload_rules<R: RuleSet>(&self, name: &str, source: &[u8]) -> Result<()>;
    pub fn rollback_rules(&self, name: &str, version: u64) -> Result<()>;
}
```

**Atomic swap via mapped memory:** Double-buffered rule region, atomic pointer swap after validation.

**Files:**
- Modify: `crates/ringkernel-core/src/hot_reload.rs` — add `RuleSet` trait + `reload_rules` method
- New: `crates/ringkernel-core/src/rules/mod.rs` — rule set trait + examples
- Keep rule parsing (OWL/SHACL) as VynGraph's responsibility — we provide the transport

### 3.4 Per-Message Provenance Metadata

**Problem:** PROV-O attribution requires tracking which agent/activity produced each message. Current envelope lacks provenance.

**Solution:** Optional `ProvenanceHeader` in `MessageEnvelope` — zero cost when unused.

**API:**
```rust
/// PROV-O inspired provenance metadata.
#[derive(Debug, Clone, RingMessage)]
pub struct ProvenanceHeader {
    /// Agent that generated this message (wasAttributedTo).
    pub agent_id: u64,
    /// Activity that produced this message (wasGeneratedBy).
    pub activity_id: u64,
    /// Parent message this was derived from (wasDerivedFrom).
    pub parent_msg_id: Option<u64>,
    /// Causal chain depth.
    pub derivation_depth: u16,
    /// Timestamp of generation (HLC).
    pub generated_at: HlcTimestamp,
}

// MessageEnvelope gets optional provenance field
pub struct MessageEnvelope {
    pub header: MessageHeader,
    pub payload: Vec<u8>,
    pub provenance: Option<ProvenanceHeader>,  // NEW
}
```

**Size cost:** 48 bytes when present, 1 byte (discriminant) when absent. Opt-in at message construction time.

**Files:**
- Modify: `crates/ringkernel-core/src/message.rs` — add provenance to envelope
- Modify: `crates/ringkernel-core/src/context.rs` — propagate provenance through K2K routing
- New: `crates/ringkernel-core/src/provenance.rs` — helpers for chain tracking

### 3.5 Multi-Tenant K2K Isolation

**Problem:** Multiple audit firms sharing a GPU cluster must not see each other's messages. Current K2K broker has no tenant concept.

**Solution:** Add `TenantId` to kernel metadata; enforce isolation at broker and GPU level.

**API:**
```rust
pub type TenantId = u64;

pub struct KernelConfig {
    // ... existing fields ...
    pub tenant_id: TenantId,
}

pub struct K2KBroker {
    // Per-tenant sub-brokers (isolation by construction)
    tenants: HashMap<TenantId, K2KSubBroker>,
}

impl K2KBroker {
    pub fn register(&self, tenant_id: TenantId, kernel_id: KernelId) -> K2KEndpoint;

    /// Cross-tenant sends are rejected at the broker level.
    pub async fn send(&self, from: &KernelId, to: &KernelId, msg: Envelope) -> Result<()>;
}
```

**Enforcement layers:**
1. **Broker level:** Reject cross-tenant sends with `K2KError::TenantMismatch`
2. **GPU level:** Tenant check in routing table lookup (rejects mismatch at device)
3. **Audit level:** Log all cross-tenant attempts to audit sink

**Files:**
- Modify: `crates/ringkernel-core/src/k2k/mod.rs` — add tenant sub-brokers
- Modify: `crates/ringkernel-core/src/k2k/broker.rs` — per-tenant message routing
- Modify: `crates/ringkernel-cuda/src/persistent.rs` — add tenant_id to routing table
- New: `crates/ringkernel-core/src/k2k/tenant.rs` — tenant registry and audit

## 4. Multi-GPU Core Infrastructure (Prerequisite)

The five gaps above require multi-GPU infrastructure. v1.1 delivers:

### 4.1 Multi-GPU Runtime

```rust
pub struct MultiGpuRuntime {
    devices: Vec<CudaRuntime>,  // one per GPU

    actor_registry: MultiGpuRegistry,

impl MultiGpuRuntime {
    pub async fn new(device_ids: &[u32]) -> Result<Self>;

    pub async fn send_cross_gpu(&self, from: &KernelId, to: &KernelId, msg: Envelope) -> Result<()>;

```

### 4.2 P2P Mapped Memory

CUDA P2P for direct GPU-to-GPU memory access when NVLink-connected:
- `cuCtxEnablePeerAccess` between device pairs
- Mapped memory visible to both GPUs
- Per-GPU K2K rings exposed via peer mapping

### 4.3 NVLink Topology Detection

```rust
pub struct NvlinkTopology {
    /// Adjacency matrix: direct_link[i][j] = bandwidth (GB/s) or 0
    pub direct_link: Vec<Vec<u32>>,
    /// Shortest hop count between GPUs
    pub hop_count: Vec<Vec<u32>>,
}

impl NvlinkTopology {
    pub fn probe() -> Result<Self>;  // via nvmlDeviceGetNvLinkState
    pub fn bandwidth(&self, from: u32, to: u32) -> u32;
    pub fn co_located_candidates(&self, gpu: u32, min_bandwidth: u32) -> Vec<u32>;
}
```

## 5. Formal Verification Plan

Target: **prove critical invariants on 2×H100 hardware before v1.1 release**.

### 5.1 Properties to Verify

| Property | Method | Hardware test |
|----------|--------|---------------|
| **No message loss during migration** | TLA+ spec + runtime assertions | 2×H100 with 100M message migration |
| **FIFO preserved per-actor across migration** | Property-based test (proptest) | 2×H100 sustained run |
| **Causal ordering (HLC) preserved cross-GPU** | Formal proof + observational test | 2×H100, concurrent K2K storm |
| **Tenant isolation: no cross-tenant messages** | Exhaustive test + audit log | Multi-tenant workload simulation |
| **Bounded latency under NVLink congestion** | Performance test with p99/p99.9 | 2×H100, saturate NVLink |
| **Actor state checksum survives migration** | CRC32 pre/post + comparison | 100 migration cycles |

### 5.2 Verification Infrastructure

**TLA+ models** (new `docs/verification/`):
- `migration.tla` — actor migration state machine (quiesce, transfer, resume)
- `multi_gpu_k2k.tla` — cross-GPU message ordering
- `tenant_isolation.tla` — tenant routing state machine

**Runtime assertions** (feature-gated `verify`):
- Invariant checks on every H2K/K2H operation
- Cross-GPU message sequence verification
- Migration checksum validation

**Benchmarks** (on 2×H100):
- Single-GPU baseline (existing H100 numbers)
- Cross-GPU K2K latency (NVLink and PCIe fallback)
- Migration latency under load
- Tenant isolation overhead (<1% target)

## 9. Risk Analysis

| Risk | Mitigation |
|------|------------|
| NVLink topology varies per VM type | Probe at runtime, fall back to PCIe if needed |
| CUDA IPC permissions on Azure | Test early; document workarounds |
| Migration race conditions | TLA+ model verifies state machine; runtime assertions catch violations |
| Cross-GPU causal ordering tricky | HLC already gives partial order; use atomics for cross-GPU barriers |
| Multi-tenant perf overhead | Design for zero-cost when tenant count = 1 |
| Hardware cost | Phase verification: dev on 1 GPU, verify on 2 GPUs only for final validation |

## 10. Open Questions

1. **PROV-O subset:** Which PROV-O relations does VynGraph actually need? (wasAttributedTo, wasGeneratedBy, wasDerivedFrom confirmed — others?)
2. **Tenant granularity:** One tenant per kernel or per message? (Recommend per-kernel for clarity, per-message for audit logs only.)
3. **Rule format:** Does VynGraph compile OWL/SHACL to a GPU-friendly representation, or send rules as opaque bytes? Affects rule validation design.
4. **Migration frequency:** Typical rate? Affects buffer sizing for in-flight message capture.
5. **Formal tools:** TLA+ sufficient, or do we want Lean/Coq proofs for the core ordering invariants?
