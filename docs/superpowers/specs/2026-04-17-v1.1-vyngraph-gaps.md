# RingKernel v1.1 — Multi-GPU + VynGraph Gap Closure

**Date:** 2026-04-17
**Goal:** Close 5 specific gaps blocking VynGraph NSAI integration + finalize multi-GPU with formal verification.
**Target hardware:** 2× H100 VMs on Azure (NVLink-connected)

---

## 1. Context

VynGraph is a neuro-symbolic AI (NSAI) reasoning system that plans to use RingKernel for GPU-accelerated inference chains. Audit firms run multi-step LLM reasoning workloads that require:
- Rebalancing actors across GPUs during long chains
- Live introspection for debugging stalled actors
- Hot-reloading inference rules (OWL/SHACL) without runtime restart
- PROV-O provenance attribution for reasoning steps
- Tenant isolation between audit firms

v1.1 closes these gaps and finalizes multi-GPU. All new GPU-level features will be verified on a 2×H100 Azure deployment with formal properties proven.

## 3. Gap Specifications

### 3.1 Multi-GPU Actor Migration (3-Phase Protocol)

**Problem:** Long NSAI inference chains may saturate one GPU while others idle. No current mechanism to migrate a live persistent actor between GPUs without losing in-flight messages.

**Expected workload (VynGraph input):**
- Steady state: 0–10 migrations/sec (graph node actors are sticky; inference rule actors long-lived)
- Rebalance burst: ~10K migrations within 30 seconds (LLM reasoning chain rebalancing)
- Peak message rate: 5M msgs/sec per actor

**Design: 3-phase protocol** (NOT flush-all-in-flight — too slow for NSAI chains).

```
Phase 1: QUIESCE (~10ms window)
  Source GPU                       Target GPU
  ──────────                       ──────────
  H2K: BeginQuiesce ──────────►
  Stop accepting new messages
  In-flight msgs → in_flight_buffer (64K slots, 6.4MB at 100B/msg)
  Emit state snapshot to mapped memory
  K2H: QuiesceComplete ────────►

Phase 2: TRANSFER (parallel with source still draining)
                                   Allocate actor slot
                                   Pre-stage routing table entries
  Transfer state via P2P ────────► Receive state (zero-copy if NVLink)
  Transfer in_flight_buffer ─────►
                                   Verify CRC32 checksum
                                   K2H: ReadyToActivate

Phase 3: SWAP + RESUME (<1ms)
  H2K: FinalizeMigration
  Broker updates routing: old_id → new_gpu  (atomic)
  In-flight messages from other actors: rerouted
                                   Resume processing
                                   Drain in_flight_buffer first, then new
                                   K2H: MigrationComplete
  Terminate source actor
```

**In-flight buffer sizing:**
- 10ms quiesce window × 5M msgs/sec = 50K msgs peak per actor
- Size buffer to **64K slots** (safety margin, power of 2)
- 6.4MB per migrating actor at 100B/msg

**Burst handling (10K concurrent migrations = 64GB):**
- **Rate limit:** Default 1K concurrent migrations
- **Disk-backed staging:** Optional buffer overflow to NVMe for ultra-bursts
- **Backpressure:** `MigrationController` refuses new migrations when buffer budget exceeded

**API:**
```rust
pub struct MigrationPlan {
    pub source_gpu: u32,
    pub target_gpu: u32,
    pub actor_id: ActorId,
    pub quiesce_window: Duration,       // default 10ms
    pub in_flight_buffer_slots: u32,    // default 65536
}

#[derive(Debug, Clone)]
pub enum RebalanceStrategy {
    /// Move least-loaded actors to under-utilized GPU (steady state).
    LoadBalance { target_imbalance: f32 },

    /// Co-locate actors that communicate frequently (NVLink-aware).
    CommunicationAware { min_nvlink_bandwidth_gbps: u32 },

    /// Explicit placement via user-provided plan.
    Explicit(Vec<MigrationPlan>),
}

pub struct MigrationControllerConfig {
    /// Maximum concurrent migrations (default 1000).
    pub max_concurrent: u32,
    /// Global in-flight buffer budget in bytes (default 8GB).
    pub global_buffer_budget: u64,
    /// Enable disk-backed staging for bursts (default false).
    pub disk_staging: Option<PathBuf>,
    /// Rate limit: max migrations per second.
    pub rate_limit_per_sec: u32,
}

impl MultiGpuRuntime {
    /// Single actor migration.
    pub async fn migrate_actor(&self, plan: MigrationPlan) -> Result<MigrationReport>;

    /// Batch migration with rate limiting and backpressure.
    pub async fn migrate_batch(
        &self,
        plans: Vec<MigrationPlan>,
    ) -> Result<Vec<MigrationReport>>;

    /// Strategy-based rebalancing.
    pub async fn rebalance(&self, strategy: RebalanceStrategy) -> Result<Vec<MigrationReport>>;
}

#[derive(Debug, Clone)]
pub struct MigrationReport {
    pub actor_id: ActorId,
    pub source_gpu: u32,
    pub target_gpu: u32,
    pub phase_durations: PhaseDurations,
    pub messages_transferred: u64,
    pub bytes_transferred: u64,
    pub used_nvlink: bool,
    pub state_checksum_match: bool,
}

pub struct PhaseDurations {
    pub quiesce: Duration,
    pub transfer: Duration,
    pub swap: Duration,
}
```

**Formal properties to verify** (TLA+ model):
- No message loss: ∀ msg sent pre-migration, msg delivered post-migration
- FIFO per-actor: message order preserved across migration
- No duplicate delivery: each message delivered exactly once
- State checksum: pre-migration checksum = post-migration checksum
- Quiesce correctness: no message accepted between `BeginQuiesce` and `FinalizeMigration`

### 3.2 Live Actor Introspection (Streaming)

**Problem:** Debugging multi-step LLM reasoning requires seeing live actor state, not periodic snapshots. Current `ActorIntrospection` is pull-only.

**Solution:** Add streaming metrics channel via K2H queue with dedicated response type.

**API:**
```rust
pub struct LiveMetrics {
    pub actor_id: ActorId,
    pub timestamp: HlcTimestamp,
    pub queue_depth: usize,
    pub inbound_rate: f64,      // messages/sec
    pub outbound_rate: f64,
    pub latency_p50: Duration,
    pub latency_p99: Duration,
    pub state_size_bytes: u64,
    pub gpu_utilization: f32,
}

impl IntrospectionStream {
    pub fn subscribe(&self, actor_id: ActorId, interval: Duration) -> Receiver<LiveMetrics>;
    pub fn subscribe_all(&self, interval: Duration) -> Receiver<LiveMetrics>;
    pub fn unsubscribe(&self, actor_id: ActorId);
}
```

**GPU-side implementation:**
- New H2K command: `SubscribeMetrics { actor_id, interval_us }`
- New K2H response: `LiveMetrics { ... }` emitted periodically from kernel
- Per-actor metrics buffered in control block (cache-aligned, no locks)

**Files:**
- Modify: `crates/ringkernel-core/src/introspection.rs` — add `IntrospectionStream`
- Modify: `crates/ringkernel-cuda/src/persistent.rs` — add metrics H2K/K2H pair
- New: `crates/ringkernel-cuda/src/cuda/metrics_collector.cu` — GPU metric emitter

**VynGraph workaround:** Emit metrics from RustGraph layer via `runtime.kernel_metrics()` polled every N ms. Acceptable but loses the sub-millisecond detail.

### 3.3 Hot Rule Reload — Compiled Artifact API

**Problem:** NSAI reasoning uses OWL/SHACL rules compiled to GPU actors. Updating rules currently requires restarting the GPU runtime.

**Design decision:** **Accept compiled artifacts, not source.** RingKernel stays rule-format-agnostic. VynGraph owns OWL/SHACL semantics, compiles to PTX via our existing codegen pipeline, hands us the artifact.

**Pipeline (VynGraph-side):**
```
OWL 2 RL source
    │
    ▼
VynGraph compiler → Rust code (via #[ring_kernel] macro)
    │
    ▼
ringkernel-cuda-codegen → PTX
    │
    ▼
ptx-cache → CompiledRule artifact
    │
    ▼ (hand to RingKernel)
HotReloadManager::reload_rule(artifact)
```

**API (RingKernel side — simple):**
```rust
/// Compiled rule artifact handed to RingKernel for hot-swap.
#[derive(Debug, Clone)]
pub struct CompiledRule {
    /// Rule set identifier (VynGraph's naming).
    pub rule_id: String,
    /// Version (monotonically increasing).
    pub version: u64,
    /// PTX for the actor kernel.
    pub ptx: Vec<u8>,
    /// Required compute capability (e.g., "sm_90").
    pub compute_cap: String,
    /// Optional dependency manifest (rule IDs this rule depends on).
    pub depends_on: Vec<String>,
    /// Signature for integrity (if signed).
    pub signature: Option<Vec<u8>>,
    /// Actor configuration.
    pub actor_config: ActorConfig,
}

pub struct ActorConfig {
    pub block_dim: (u32, u32, u32),
    pub grid_dim: (u32, u32, u32),
    pub shared_mem_bytes: u32,
    pub max_in_flight: u32,
}

impl HotReloadManager {
    /// Register a new rule with the runtime (first-time load).
    pub async fn register_rule(&self, rule: CompiledRule) -> Result<RuleHandle>;

    /// Hot-swap an existing rule atomically. Validates:
    /// - Signature (if configured)
    /// - Compute capability matches device
    /// - Dependencies are present
    /// - Version is monotonically newer
    pub async fn reload_rule(&self, rule: CompiledRule) -> Result<ReloadReport>;

    /// Rollback to a prior version (must be in version history).
    pub async fn rollback_rule(&self, rule_id: &str, to_version: u64) -> Result<()>;

    /// List all registered rules with versions.
    pub fn list_rules(&self) -> Vec<RuleMetadata>;
}

pub struct ReloadReport {
    pub rule_id: String,
    pub from_version: u64,
    pub to_version: u64,
    pub quiesce_duration: Duration,
    pub swap_duration: Duration,
    pub messages_in_flight_during_swap: u64,
}
```

**Hot-swap protocol (atomic from application perspective):**
1. **Validate** artifact (signature, compute cap, deps)
2. **Pre-stage** new PTX on device (new kernel loaded, not yet active)
3. **Quiesce** old actor (H2K quiesce command, drain in-flight)
4. **Atomic pointer swap** in device-side actor registry
5. **Resume** with new kernel; old kernel terminates after grace period
6. **Audit log** the transition

**Guarantees:**
- No message loss (in-flight drained before swap)
- Version monotonicity (rejects downgrade unless explicit rollback)
- Rollback always available (keeps N prior versions)
- Failed validation = no state change

**Files:**
- Modify: `crates/ringkernel-core/src/hot_reload.rs` — extend with rule APIs
- New: `crates/ringkernel-core/src/rules/mod.rs` — `CompiledRule`, `ActorConfig`, `ReloadReport`
- New: `crates/ringkernel-core/src/rules/registry.rs` — rule registry, version history
- Modify: `crates/ringkernel-cuda/src/persistent.rs` — atomic actor swap support

**What stays in VynGraph:** OWL parsing, SHACL parsing, rule semantics, compilation logic. We just execute the compiled artifacts.

### 3.4 Per-Message Provenance Metadata (PROV-O subset)

**Problem:** PROV-O attribution requires tracking agent/activity/entity relationships for each message. Needed for:
- "fraud alert attributed to `fraud_triangle_analytic`"
- "going-concern informed by cash-flow analysis"
- "LLM generation associated with auditor prompt"
- Delegation chains (assistant acted on behalf of partner)

**Solution:** Optional `ProvenanceHeader` in `MessageEnvelope` supporting 8 PROV-O relations. Zero cost when unused.

**PROV-O relations in v1.1:**

| Relation | Types | VynGraph usage |
|----------|-------|----------------|
| `wasAttributedTo` | Entity → Agent | fraud alert ← fraud_triangle_analytic |
| `wasGeneratedBy` | Entity → Activity | consolidated_balance ← GAAP_consolidation_run |
| `wasDerivedFrom` | Entity → Entity | source-trace lineage for audit evidence |
| `used` | Activity → Entity | SoD check consumed journal entries |
| `wasInformedBy` | Activity → Activity | going-concern informed by cash-flow analysis |
| `wasAssociatedWith` | Activity → Agent | LLM generation associated with auditor prompt |
| `actedOnBehalfOf` | Agent → Agent | assistant acted on behalf of partner |
| Plan (Entity subclass) | n/a | audit programs as plans |

**Deferred to v1.5:** Qualified relations (qualifiedAttribution, qualifiedDerivation, etc.) — HLC already provides temporal qualification we need for v1.1.

**API:**
```rust
/// PROV-O subset — 8 core relations for NSAI reasoning chains.
#[derive(Debug, Clone, RingMessage)]
pub struct ProvenanceHeader {
    /// Node classification in PROV graph.
    pub node_type: ProvNodeType,

    /// Globally unique PROV node ID (Entity, Activity, or Agent ID).
    pub node_id: u64,

    /// Direct PROV relations from this node. Indexed by PROV relation type.
    /// Empty slot = no relation. 4 slots = most messages fit without overflow.
    pub relations: [ProvRelation; 4],

    /// If more than 4 relations, overflow link to extended record (off-band).
    pub overflow_ref: Option<u64>,

    /// HLC timestamp of PROV event (generation/attribution time).
    pub prov_timestamp: HlcTimestamp,

    /// Optional plan ID (if this activity follows a Plan).
    pub plan_id: Option<u64>,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ProvNodeType {
    Entity,
    Activity,
    Agent,
    Plan,   // Entity subclass, explicit for audit programs
}

#[derive(Debug, Clone, Copy)]
pub struct ProvRelation {
    pub kind: ProvRelationKind,
    pub target_id: u64,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
#[repr(u8)]
pub enum ProvRelationKind {
    None = 0,
    WasAttributedTo = 1,      // Entity → Agent
    WasGeneratedBy = 2,       // Entity → Activity
    WasDerivedFrom = 3,       // Entity → Entity
    Used = 4,                 // Activity → Entity
    WasInformedBy = 5,        // Activity → Activity
    WasAssociatedWith = 6,    // Activity → Agent
    ActedOnBehalfOf = 7,      // Agent → Agent
}

// MessageEnvelope gets optional provenance
pub struct MessageEnvelope {
    pub header: MessageHeader,
    pub payload: Vec<u8>,
    pub provenance: Option<ProvenanceHeader>,  // NEW
    pub tenant_id: TenantId,                   // NEW (see §3.5)
    pub audit_tag: AuditTag,                   // NEW — org_id + engagement_id
}
```

**Size cost:**
- Absent: 1 byte discriminant
- Present: ~88 bytes (ProvenanceHeader with 4 inline relations)
- Overflow cases (>4 relations): one extra cache line via `overflow_ref`

**Design notes:**
- Inline 4 relations cover 95%+ of messages (most NSAI steps have 1-3 PROV edges)
- Overflow avoids bloating the common case
- `Plan` as Entity subclass (not separate type) — matches PROV-O spec
- All IDs are `u64` — VynGraph controls the ID namespace

**Files:**
- Modify: `crates/ringkernel-core/src/message.rs` — add provenance to envelope
- Modify: `crates/ringkernel-core/src/context.rs` — propagate provenance through K2K routing
- New: `crates/ringkernel-core/src/provenance.rs` — types, builder, chain validation helpers

### 3.5 Multi-Tenant K2K Isolation (Two-Tier)

**Problem:** Multiple audit firms sharing a GPU cluster must not cross-contaminate. Audit firms also run multiple concurrent engagements per tenant — need quota accounting per engagement.

**Solution:** Two-tier tenancy model:
1. **Per-kernel `tenant_id`** = primary security boundary (routing, quotas, isolation)
2. **Per-message `tenant_id` + `audit_tag`** = audit trail / observability
3. **Two-level hierarchy** = `org_id` (audit firm) + `engagement_id` (specific audit)

**Security by construction:**
- Tenant ID stamped at kernel launch, immutable for kernel lifetime
- Routing table lookups enforce tenant match at both broker and GPU level
- Cross-tenant attempts rejected with `K2KError::TenantMismatch` and audit-logged

**API:**
```rust
pub type TenantId = u64;

/// Two-level hierarchy for audit firm workload accounting.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct AuditTag {
    /// Audit firm / organization ID.
    pub org_id: u64,
    /// Specific engagement / audit project ID.
    pub engagement_id: u64,
}

/// Per-tenant quota and cost tracking.
#[derive(Debug, Clone)]
pub struct TenantQuota {
    pub max_concurrent_kernels: u32,
    pub max_gpu_memory_bytes: u64,
    pub max_messages_per_sec: u64,
    /// Per-engagement cost tracking (billable GPU-seconds).
    pub per_engagement_budget: HashMap<u64, Duration>,
}

pub struct KernelConfig {
    // ... existing fields ...
    pub tenant_id: TenantId,        // Immutable security boundary
    pub audit_tag: AuditTag,        // For cost/audit accounting
}

pub struct K2KBroker {
    /// Per-tenant sub-brokers — isolation by construction (no shared routing state).
    tenants: HashMap<TenantId, K2KSubBroker>,
    /// Tenant registry with quotas.
    registry: Arc<TenantRegistry>,
}

pub struct TenantRegistry {
    tenants: RwLock<HashMap<TenantId, TenantInfo>>,
    audit_sink: Arc<dyn AuditSink>,
}

impl TenantRegistry {
    pub fn register(&self, tenant_id: TenantId, quota: TenantQuota) -> Result<()>;
    pub fn check_quota(&self, tenant_id: TenantId, audit_tag: AuditTag) -> Result<()>;
    pub fn track_usage(&self, tag: AuditTag, gpu_seconds: Duration);
    pub fn get_engagement_cost(&self, tag: AuditTag) -> Duration;
}

impl K2KBroker {
    pub fn register(
        &self,
        tenant_id: TenantId,
        audit_tag: AuditTag,
        kernel_id: KernelId,
    ) -> K2KEndpoint;

    /// Cross-tenant sends rejected at broker level (no routing table entry).
    /// Same-tenant sends stamp audit_tag into envelope.
    pub async fn send(&self, from: &KernelId, to: &KernelId, msg: Envelope) -> Result<()>;
}
```

**Enforcement layers:**
1. **Broker level:** Cross-tenant sends have no route in the per-tenant sub-broker — rejected with `K2KError::TenantMismatch { from: t1, to: t2 }`
2. **GPU level:** Routing table entries stamped with `tenant_id`; device-side mismatch = no message delivery + error counter incremented
3. **Envelope level:** Every message carries `tenant_id` + `audit_tag` for replay/audit
4. **Audit level:** All cross-tenant attempts logged to audit sink with full context

**Performance:**
- Zero-cost when tenant count = 1 (sub-broker fast path, no HashMap lookup)
- Per-tenant sub-broker adds one HashMap lookup (~20ns) in broker send
- GPU-side tenant check adds one `u64` compare + branch (~1 cycle)
- Target: <1% overhead vs single-tenant baseline

## 5. Formal Verification Plan

Target: **prove critical invariants on 2×H100 hardware before v1.1 release**.

### 5.1 Properties to Verify

| Property | Method | Hardware test |
|----------|--------|---------------|
| **No message loss during migration** | TLA+ spec + runtime assertions | 2×H100 with 100M message migration |
| **FIFO preserved per-actor across migration** | Property-based test (proptest) | 2×H100 sustained run |
| **Causal ordering (HLC) preserved cross-GPU** | Formal proof + observational test | 2×H100, concurrent K2K storm |
| **Tenant isolation: no cross-tenant messages** | Exhaustive test + audit log | Multi-tenant workload simulation |
| **Bounded latency under NVLink congestion** | Performance test with p99/p99.9 | 2×H100, saturate NVLink |
| **Actor state checksum survives migration** | CRC32 pre/post + comparison | 100 migration cycles |

### 5.2 Verification Infrastructure

**Tool choice: TLA+ only.** Bounded model checking covers ~95% of protocol invariants at low cost. TLAPS (refinement proofs) available if needed. No Lean/Coq in RingKernel — those belong in VynGraph's NSAI compliance layer (v1.5 scope).

**TLA+ models** (new `docs/verification/`):
- `hlc.tla` — HLC monotonicity and causal ordering invariants
- `k2k_delivery.tla` — K2K delivery semantics (single-GPU baseline)
- `migration.tla` — 3-phase migration state machine (quiesce, transfer, swap)
- `multi_gpu_k2k.tla` — cross-GPU message ordering with NVLink
- `tenant_isolation.tla` — tenant routing state machine (no cross-tenant leak)
- `actor_lifecycle.tla` — create/destroy/restart invariants

**TLC model checker runs:**
- Bounded: 3 GPUs × 5 actors × 100 messages (fast, catches most bugs)
- Extended: 2 GPUs × 10 actors × 10000 messages (overnight CI job)

**Runtime assertions** (feature-gated `verify`):
- HLC monotonicity checks on every tick
- Message sequence number verification (K2K FIFO)
- Migration checksum validation (CRC32 pre/post)
- Tenant ID match on every routing decision
- Provenance chain consistency (no cycles, depth bounded)

**Benchmarks** (on 2×H100):
- Single-GPU baseline (must match existing H100 numbers — 8,698x, 5.54M ops/s)
- Cross-GPU K2K latency (NVLink and PCIe fallback)
- Migration latency: steady state + burst
- Tenant isolation overhead (<1% target vs single-tenant baseline)
- Provenance overhead (<2% target vs no-provenance baseline)

**Future: Lean 4 compliance certificates** — strictly out of scope for v1.1. If VynGraph needs them (for "this audit program covers ISA 315" unbounded proofs), we'd add a `formal-proofs` feature flag with optional Lean toolchain dependency in v1.5. Critical path stays TLA+.

## 9. Risk Analysis

| Risk | Mitigation |
|------|------------|
| NVLink topology varies per VM type | Probe at runtime, fall back to PCIe if needed |
| CUDA IPC permissions on Azure | Test early; document workarounds |
| Migration race conditions | TLA+ model verifies state machine; runtime assertions catch violations |
| Cross-GPU causal ordering tricky | HLC already gives partial order; use atomics for cross-GPU barriers |
| Multi-tenant perf overhead | Design for zero-cost when tenant count = 1 |
| Hardware cost | Phase verification: dev on 1 GPU, verify on 2 GPUs only for final validation |

## 10. Design Decisions (Resolved 2026-04-17 with VynGraph team)

| # | Question | Resolution |
|---|----------|------------|
| 1 | PROV-O subset | **8 core relations** in v1.1: `wasAttributedTo`, `wasGeneratedBy`, `wasDerivedFrom`, `used`, `wasInformedBy`, `wasAssociatedWith`, `actedOnBehalfOf`, `Plan` entity subclass. Qualified relations deferred to v1.5 (HLC provides temporal qualification). |
| 2 | Tenant granularity | **Two-tier:** per-kernel `tenant_id` as security boundary (routing, quotas), per-message `tenant_id + audit_tag` for audit trail. Also `AuditTag { org_id, engagement_id }` for audit firm accounting (multiple engagements per tenant). |
| 3 | Rule format | **Compiled artifact API.** VynGraph compiles OWL 2 RL → Rust → PTX via our existing `ringkernel-cuda-codegen`. RingKernel accepts `CompiledRule` (PTX + metadata + signature) and hot-swaps atomically. We stay rule-format-agnostic. |
| 4 | Migration frequency | Steady: 0-10/sec. Burst: 10K/30s. **3-phase protocol** (quiesce/transfer/swap), 64K in-flight buffer per actor (~6.4MB at 100B/msg), default rate limit 1K concurrent, optional disk-backed staging for bursts. |
| 5 | Formal tools | **TLA+ only.** TLC bounded model checking for protocol invariants; TLAPS available if refinement proofs needed. No Lean/Coq — those are VynGraph's problem (v1.5 NSAI compliance certificates), not kernel runtime. |
