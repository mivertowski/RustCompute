## Headline numbers

### Experiment 1 — K2K tier latency

Intra-cluster round-trip (host-observed), 1000 trials per cell, 200
warmup iterations, H100 NVL. Values are **mean ± 95% CI**. All three
tiers are measured directly via `paper_tier_latency` (no proxy):

| Payload | SMEM (ns)   | DSMEM (ns)   | HBM (ns)    | HBM:SMEM ratio |
|--------:|------------:|-------------:|------------:|---------------:|
|    64 B | 6,793 ± 41  |  9,012 ± 40  | 10,460 ± 39 | 1.54× |
|   256 B | 6,702 ± 31  | 10,598 ± 34  | 12,066 ± 33 | 1.80× |
|  1 KiB  | 6,722 ± 34  | 14,059 ± 34  | 15,811 ± 145| 2.35× |
|  4 KiB  | 6,749 ± 43  | 15,289 ± 106 | 18,237 ± 30 | 2.70× |

SMEM latency is payload-independent (lives entirely in each SM's
shared memory); DSMEM scales with payload; HBM adds the global
memory round-trip — clean monotonic SMEM < DSMEM < HBM hierarchy
across every payload size. See `fig:tier-latency`.

The HBM tier is exercised by the new `cluster_hbm_k2k` kernel
(cross-cluster K2K via global memory with `grid.sync()`) added in
v1.1 specifically for this measurement.

### Experiment 3 — Lifecycle rule overhead

End-to-end issue-to-ack ns for each SOS rule, 500 trials per rule:

| Rule       | mean (ns) | stddev | p99 (ns) |
|-----------:|----------:|-------:|---------:|
| Spawn      | 23.1      | 4.7    | 30       |
| Activate   | 23.4      | 4.7    | 30       |
| Quiesce    | 23.8      | 4.8    | 30       |
| Terminate  | 23.7      | 4.8    | 30       |
| Restart    | 23.1      | 4.6    | 30       |

All five rules cluster around ~23 ns mean / 30 ns p99 — flat within
noise, consistent with the paper's §4 claim of sub-microsecond rule
application.

### Experiment 4 — Sustained throughput (60 s × 4 trials)

Data in `exp4_sustained_timeseries/trial{1,2,3,4}.log` (one window
every 5 s during each 60 s run). Per-trial aggregates:

- Throughput: ~5.1 M ops/s sustained on the SPSC queue path
- Latency p50 / p95 / p99: **90 / 100 / 110 ns**, flat across all
  12 × 4 = 48 windows (no drift, no thermal throttle).
- ECC error deltas pre → post: 0 corrected, 0 uncorrected per GPU.

### Experiment 6 — NVLink P2P vs host-stage migration

Full results in `exp6_nvlink_migration/migration.csv`. Per-path
p99 latency (200 trials) and P2P:host ratio:

| Size    | P2P p99 (ns)   | Host p99 (ns)  | P2P speedup (mean) |
|--------:|---------------:|---------------:|-------------------:|
|  1 KiB  |          8,029 |         13,840 | 1.96× |
|  4 KiB  |          7,850 |         16,770 | 2.02× |
| 64 KiB  |          8,180 |         20,120 | 2.44× |
|  1 MiB  |         12,630 |         57,510 | 5.08× |
| 16 MiB  |         71,439 |        603,605 | 8.65× |

At 16 MiB, NVLink P2P is **8.7× faster** in mean and **8.4× faster**
in p99 than host-staged migration. See `fig:migration-cost`.

### Experiment 6b — Multi-GPU K2K sustained bandwidth (v1.1 addendum)

Back-to-back `cuMemcpyPeerAsync` bursts of 256 transfers on a single
stream (32 warmup) — fills the NVLink bundle:

| Size    | Total (ns) | Bandwidth (GB/s) | Peak utilization |
|--------:|-----------:|-----------------:|-----------------:|
|  4 KiB  |    455,097 |            2.30 |   0.7 % |
| 64 KiB  |    535,035 |           31.36 |   9.9 % |
|  1 MiB  |  1,500,029 |          178.95 |  56.3 % |
| 16 MiB  | 16,661,095 |          257.78 |  81.1 % |

Theoretical peak for a 12-link H100 NVL bundle (26.562 GB/s per
link, unidirectional) is ~318 GB/s. At 16 MiB we reach 81 % of
peak — the stream is saturating the NVLink fabric. Data:
`docs/paper/experiments/results/<ts>/exp6b_mgpu_bw/mgpu_bw.csv`.

CPU-path tests are feature-complete and pass on 1 or 2 GPUs.

- `tenant_stats_reports_per_tenant_counts`
- `quota_enforcement_rejects_over_rate_limit`
- `re_registering_kernel_moves_it_between_tenants`
- `legacy_single_tenant_send_unchanged`
- `send_with_audit_overrides_registration_tag`
- `engagement_cost_accumulates_across_sends`
- `engagement_cost_separate_across_audit_tags`

**Cross-tenant leak count: 0** across every rejection path.

### Provenance header & chain (Gap 2)

Test: `cargo test -p ringkernel-core --release provenance`

22 subtests, all pass. Covers all 8 PROV-O relation kinds, chain
walk depth bound, cycle detection, signature validation, and
envelope-attached header round-trip.

### Hot rule reload (Gap 3)

Test: `cargo test -p ringkernel-core --release rules::registry::tests`

32 subtests, all pass. Covers version bump rejection policy
(equal/lower versions rejected, higher versions accepted),
signature verification, rollback semantics, quiescence of
in-flight evaluators, and dependency compilation.

### Live introspection streaming (Gap 4)

Covered by the broader `ringkernel-core` test suite (streaming
iterator + EWMA decay + drop-tolerant ring buffer). No cross-tenant
leaks surfaced in stress tests.

## v1.1 Additions Beyond the Original Spec

Four items that the v1.1 handover marked as "deferred" were addressed
in this release:

### Incremental / delta checkpoints

`Checkpoint::delta_from(base, new)` returns a checkpoint containing
only chunks whose `(type, id)` data differs from `base`, plus chunks
new in `new`. `Checkpoint::applied_with_delta(base, delta)` restores
the full checkpoint and verifies the recorded parent digest
(`DELTA_PARENT_DIGEST_KEY`) matches the supplied base. 5 new tests
under `checkpoint::tests::delta_*` and `content_digest_*`.

### Intra-block warp work stealing

`warp_work_steal` CUDA kernel demonstrates warps atomically stealing
tasks from a shared counter; lane-0 tallies per warp let the host
audit that every task is processed exactly once. Two integration
tests in `tests/warp_work_steal.rs` verify correctness and
load-distribution properties. Intra-cluster (DSMEM) and cross-
cluster (HBM) stealing remain future work.

### HBM tier direct measurement

`cluster_hbm_k2k` kernel (cross-cluster K2K via global memory with
`grid.sync()`) replaces the snapshot/restart proxy data used in the
initial v1.1 run. Experiment 1 now measures all three K2K tiers
directly.

### Multi-GPU K2K sustained bandwidth micro-bench

New `paper_multi_gpu_k2k_bw` test + `exp6b_mgpu_bw/mgpu_bw.csv`
artifact. Complements Experiment 6 (one-shot latency) with
sustained bandwidth measurements at 4 sizes.

## Reproducibility

Each experiment's `run.sh` is idempotent; results are captured to
`docs/paper/experiments/results/<timestamp>/` with an accompanying
`manifest.json` recording commit, driver, CUDA, Rust, and GPU info.
The top-level `run_all.sh` chains all six experiments.

For ad-hoc reruns:

```bash
cd docs/paper/experiments
./run_all.sh                 # full 6-experiment pipeline
./run_all.sh --skip=4,6       # skip long or 2-GPU experiments
./run_all.sh --archive-only   # just collect existing results
```

`--archive-only` is useful for re-packaging after a partial run.
